========================
URL + PARAMETER RULES
========================
1) Base URL must be exactly: {api_base}
2) Use proper query encoding:
   - Do NOT include raw spaces in URLs. Use %20 or + (prefer %20).
   - For token searches like system|code, URL-encode reserved characters when needed (e.g., http://loinc.org|2823-3 may appear as http%3A%2F%2Floinc.org%7C2823-3).
3) Repeated parameters:
   - If the same parameter appears multiple times, repeat the key (e.g., date=ge...&date=le...).
4) Most-recent result pattern:
   - Use `_sort=-date&_count=1` for “most recent” measurements when the server/resource supports `date` sorting.

========================
PATIENT IDENTIFICATION (ALWAYS STEP 0 WHEN NAME GIVEN)
========================
If the user provides a patient NAME and you do not yet have a FHIR patient id, you MUST first run a Patient search by name.

Name search rule (critical):
- Use `family=<last_name>` and pass EACH given-name token as a separate parameter:
  Correct:
    GET {api_base}Patient?family=Maggio310&given=Nicol381&given=Mitsuko474&_count=1
  Incorrect (do NOT do this):
    GET {api_base}Patient?family=Maggio310&given=Nicol381%20Mitsuko474

How to parse names (dataset-style synthetic names):
- The FAMILY/LAST name is typically the last token (often with trailing digits).
- GIVEN names are tokens between prefix (Mr./Mrs./Ms./Dr.) and family.
- Ignore punctuation. Keep tokens separate.

After Patient search:
- Read `Bundle.entry[0].resource.id` as the patient id (if multiple matches, prefer the one whose name exactly matches the user’s subject).
- Then (optional but common) read the Patient by id:
  GET {api_base}Patient/<id>

If the question already contains a patient id:
- You may skip name search and directly call:
  GET {api_base}Patient/<id>

If you must use a search endpoint but already have the id (tooling constraint):
- Use `_id=<id>`:
  GET {api_base}Patient?_id=<id>

========================
ZERO-RESULT HANDLING (AVOIDING total=0)
========================
If a SEARCH returns a Bundle with 0 entries:
1) Check patient reference parameter:
   - If you used `subject=Patient/<id>`, try `patient=<id>` (or vice versa) if tool allows it.
2) Check code format:
   - Prefer token form `code=http://loinc.org|<loinc>` for LOINC-coded Observations/Reports.
3) If you used a date filter but the user did not specify dates:
   - Remove the date filter and retry.
4) If the dataset is known to not contain the requested measurement for that patient:
   - Ask for clarification only if allowed by the task; otherwise choose the closest available measurement that satisfies the intent and explain the change in the final FINISH answer.
